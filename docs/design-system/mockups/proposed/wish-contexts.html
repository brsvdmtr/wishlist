Контексты желаний
Теги · Поводы · Аудитории · Подборки · Витрины
Мокапы фичи организации желаний поверх существующих вишлистов.
1. Ключевые продуктовые решения
До того как рисовать экраны, нужно договориться о нескольких развилках — они влияют и на UX, и на данные. Ниже — рекомендация для каждой. Если что-то выглядит не так, скажи — и я перерисую.
Важный терминологический конфликт.
В продукте уже заняты слова «витрина» (Profile Showcase — секция профиля с закреплёнными вишлистами, существует с 2026-04-14) и «подборка» (Curated Selection — временная ссылка на 45 дней для отправки N желаний). Новая фича вводит ещё одно значение «подборки» и «витрины». Ниже — мой вариант развести термины без переписывания существующего.
Решение 1 · терминология
Как назвать Collection и Showcase, не ломая текущие термины
Предлагаю: Collection → «Подборка» (постоянная, кросс-вишлистная). Showcase → «Витрина» (публичная shareable-страница). Существующая Curated Selection переименовывается в UI в «Ссылка-подборка» или «Быстрая подборка» и становится частным случаем шаринга Collection. Существующий Profile Showcase из UserProfile → «Обложка профиля». В коде модели не переименовываем — меняем только тексты UI.
Collection = Подборка Showcase = Витрина Curated → «Быстрая подборка» Profile Showcase → «Обложка профиля»
Решение 2 · теги
Ломать или не ломать существующую модель Tag
Сейчас Tag привязан к wishlistId — тег живёт в одном вишлисте. Для фичи нужен тег уровня пользователя (чтобы один тег работал через все вишлисты). Рекомендую: добавить новую таблицу UserTag (owner-scoped), связь через WishTag. Старый Tag оставить для внутренней механики wishlist-категорий / не трогать. Мигрировать постепенно: в UI показывать только UserTag, старый Tag маскируется.
Новая таблица UserTag Старый Tag не трогаем Без деструктивной миграции
Решение 3 · поводы
Occasion ≠ GiftOccasion (они не должны путаться)
В проекте уже есть GiftOccasion — это события конкретных людей из Gift Calendar («ДР Маши 5 июля»). Новый Occasion — это справочная метка, вешается на желание: «это подходит на ДР / НГ / свадьбу». Рекомендую: завести новую модель Occasion как справочник (system-seeded), связь с wish через WishOccasion. GiftOccasion не трогаем. В UI называю «повод» — в проекте это свободное слово, конфликта нет.
Новый Occasion (справочник) GiftOccasion не трогаем Риск путаницы у юзера — смягчается копирайтом
Решение 4 · множественность
Сколько поводов и аудиторий можно повесить на желание
Рекомендация: несколько поводов и несколько аудиторий — так пользователь реально думает. Mutual exclusivity реализуется двумя «wildcard» значениями: «без повода» и «любой повод» исключают все остальные поводы; «кто угодно» исключает конкретные аудитории. UI блокирует несовместимые выборы с подсказкой, а не выдаёт ошибку постфактум.
Multiple occasions Multiple audiences Wildcard исключает остальное
Решение 5 · режимы витрин
Manual vs Dynamic vs Filter — и что делать при поломке источника
Поддерживаем три режима: Manual (ручной список wishes), Dynamic Collection (зеркалит содержимое подборки), Dynamic Filter (правило: тег + повод + аудитория). Если источник исчезает — виртрина становится broken-config, owner видит баннер с CTA «Починить», публичный зритель видит то, что удалось зарезолвить; если пусто — empty state «Некоторые желания больше не показываются».
Manual Dynamic Collection Dynamic Filter Broken-config UI для владельца
Решение 6 · приватность шаринга
Может ли Collection быть публичной сама по себе
Рекомендация: Collection не шарится напрямую как публичная страница. Если пользователь хочет поделиться — он создаёт Showcase поверх этой Collection. Это убирает двойную shareable-поверхность и делает Collection чисто организационным слоем. Исключение: «быстрая подборка» (старый CuratedSelection) остаётся как самостоятельный инструмент ad-hoc шаринга и живёт рядом.
Collection = private layer Showcase = public layer Curated = ad-hoc quick share
Решение 7 · архив и приватность
Что показывает публичная Витрина при частично недоступном контенте
Правило: архивные / приватные / удалённые wishes выпадают из публичной витрины молча. Если пропадает часть — витрина рендерит то, что есть, плюс тонкая подпись: «Некоторые желания больше не показываются». Если пусто — empty state, не ошибка. Broken-config (удалён источник) — owner видит предупреждение, public видит то же empty state.
Graceful degradation Silent skip Тонкая подпись для прозрачности
Решение 8 · Pro vs Free
Границы плана
Free: теги до 10, подборок до 3 (private only), витрин 0. Закрепление в профиле — нет. Кастомная обложка витрины — нет.
Pro: теги unlimited, подборок unlimited, витрин до 10, cover/описание/закрепление в профиле до 5 витрин/подборок. Аналитика по витринам. Dynamic filter mode.
Free: 10 tags / 3 коллекций / 0 витрин Pro: ∞ / ∞ / 10 витрин
Решение 9 · синхронизация
Dynamic Filter — snapshot или live
Рекомендация: dynamic filter всегда разрешается live при рендере публичной страницы. Без снапшотов. Плюс: пользователь меняет wish — публика видит сразу. Минус: удаление тега ведёт к пустой витрине — решается CTA «Перенастроить» на стороне владельца. Для тяжёлых витрин — кешируем result set на короткое время (5–10 минут) на backend.
Live resolve Короткий cache Без снапшотов
Решение 10 · комментарии, брони
Ничего не ломаем в существующих механиках
Бронь и комментарии остаются на уровне wish. Витрина и подборка — только ракурсы. Reply, reserve, replies-on-comments, privacy по существующим правилам. Точка входа «открыть wish из витрины» ведёт в ту же карточку, что из вишлиста (тот же компонент, тот же actorHash). Никаких параллельных веток комментариев и броней внутри подборок/витрин.
Wish — единственный источник Никаких parallel threads
Вопросы, которые я хочу услышать от тебя
1. Термины — согласен на «Подборка / Витрина / Быстрая подборка / Обложка профиля»? Или хочешь другой расклад?
2. Free-лимиты — 0 витрин на Free или всё же одна минимальная?
3. «Быстрая подборка» (существующий CuratedSelection) — сохраняем рядом или прячем за Collection-шарингом?
4. «Без повода / Любой повод» — обе wildcard-опции или только одна (рекомендую обе, но это спорно)?
5. Cover у Collection — только у Showcase, или у Collection тоже (в профиле она видна — значит обложка нужна)?
6. «Ссылка-подборка» с TTL 45 дней — сохранять у нового Showcase по умолчанию, или Showcase только постоянные без TTL?

Bulk-bar внизу скроллится горизонтально. «В подборку» — синий (основное действие), остальные — обычный surface. Каждое действие открывает тот же sheet, что в редакторе одного желания, но применяется ко всем выбранным.

Посмотреть как гость
Ключевой privacy-контракт: внутри витрины владелец видит маркировку «не показывается гостям». Гость получает отфильтрованный список без упоминания скрытого. Precision over transparency — владелец знает, что скрыто, гость не получает утечку.

Текущие подборки отмечены. Можно снять или добавить — всё как список.
Нет отдельной ошибки «уже добавлено». Состояние «уже здесь» — это просто включённый чекбокс с подписью. Снимаешь — удаляется из подборки, ставишь — добавляется. Идемпотентно.
Чего в мокапах нет, но обязательно в реализации:
· конфликт аудиторий «кто угодно» ↔ конкретные — тот же pattern, что у поводов
· смена приватности вишлиста → пересчёт видимости в витринах (cron / on-save trigger)
· витрина с 0 видимых wishes → публичный empty state, счётчик не показывается
· попытка открыть старую ссылку (regenerated slug) → link_revoked state (см. секция 6-Г)
· удаление тега, используемого в 3 dynamic collections → confirm-sheet с перечислением затронутых
· несовместимые bulk actions (назначить «Любой повод» на wishes, где уже есть конкретный) → toast с результатом: «5 обновлено, 3 пропущено»
