Общее желание в нескольких wishlist'ах
Одна каноничная сущность — множество размещений. Изменения карточки, бронь, комментарии и статус синхронизируются во всех wishlist'ах, где желание размещено. Порядок и категория — локальные.
Canonical Wish × N Placements
1. Концепция модели
Желание — это сущность пользователя. Wishlist — контейнер с размещениями (placement'ами). Одно желание может одновременно жить в нескольких wishlist'ах, оставаясь одной сущностью, а не копией.

Общее во всех wishlist'ах
Эти поля и состояния живут на уровне wish. Изменил в одном месте — применилось везде.
  • Название, описание
  • Ссылка, фото, цена, валюта
  • Статус (активно / куплено / архив)
  • Бронь (резерв, TTL, освобождение)
  • Комментарии и reply — один общий тред
  • Marketplace-данные (refresh, sourceType)
  • Архивация, удаление
📂
Локальное для каждого wishlist
Это свойства размещения, а не желания. В разных wishlist'ах могут отличаться.
  • Порядок (position в списке)
  • Категория внутри wishlist
  • Локальный pin / highlight
  • (резерв на будущее) placement-specific note
Ключевые принципы
🎯
Одно желание — одна сущность
Нет «копий с синком». Есть одна каноничная Wish-запись. UI никогда не создаёт shared-behavior через бэкграунд-синхронизацию.
🔒
Privacy по контексту входа
Публичный зритель, пришедший через public-wishlist A, никогда не узнаёт, что то же желание лежит в private-wishlist B того же владельца.
🧬
«Копия» — отдельное действие
Если пользователь хочет независимую карточку — есть явное «Создать отдельную копию». Это другой сценарий, не дефолт.
🚫
Нет orphan-wishes
У каждой Wish должен быть хотя бы один placement. Убрал последнее размещение — желание удаляется полностью (после явного подтверждения).

Privacy-инварианты
API-правила:
1. Публичные endpoint'ы никогда не возвращают wish.placements[] целиком — только одну запись контекста.
2. Счётчик placement'ов в публичном ответе либо отсутствует, либо считает только public-placements.
3. Комментарий без wishlistId контекста не должен быть публично адресуемым (никаких deeplink'ов через private-placement).
4. Owner-check перед любым ответом, включающим management-информацию (placements list, edit screen).

12. Архитектурные решения
Non-negotiable решения перед имплементацией. Каждое — явный выбор с указанием альтернатив. Эти решения зафиксируют модель и сохранят консистентность фичи.
Решение 1
Canonical Wish + WishlistWishPlacement
Wish — каноничная сущность, принадлежит owner user. Wishlist содержит размещения (join-record'ы). Никаких копий с фоновой синхронизацией.
Wish ✕ Placement
✕ shadow-копии
✕ background sync
Решение 2
Глобальные vs локальные поля
Global (wish): title, description, url, images, price, currency, status, reservation, comments, archive.
Local (placement): position, categoryId, pinnedInWishlist.
category = placement-level
position = placement-level
Решение 3
Last placement → require explicit full-delete confirm
Убрать последнее размещение нельзя без явного подтверждения. API: DELETE /placements/:id возвращает 409 requires_full_delete_confirmation, если это последний placement. Клиент показывает соответствующий dialog и отправляет DELETE /wishes/:id.
409 + separate delete
✕ implicit cascade
Решение 4
Full-delete разрешён при активной брони
Следуем текущей политике удаления wish. Усиливаем confirm: показываем «Желание уже забронировано», но не блокируем. Бронь — не contract.
allow + strong warn
✕ hard block
Решение 5
Privacy по context-entry
Доступ к wish публичному зрителю определяется через public-wishlist, в котором он сейчас смотрит. Никаких «wish existed elsewhere» → ответ 200. Backend filters: resolvePublicWish({wishId, viaWishlistSlug}) → 403 or wish-in-context.
context-scoped
✕ global-visible
Решение 6
Категория — placement-level
Category хранится на placement: WishlistWishPlacement.categoryId. Одно и то же желание может быть в «Электронике» в одном wishlist'е и «Технике» в другом.
placement.categoryId
Решение 7
Порядок — placement-level
WishlistWishPlacement.position. Reorder внутри wishlist'а меняет только позиции placement'ов в этом wishlist'е. Index: (wishlistId, position).
placement.position
Решение 8
Migration: 1 wish → 1 placement
Все существующие wishes получают ровно один placement с текущим wishlistId. Существующие комментарии/бронь остаются на wish. Category/position переезжают на placement. Deploy: schema → backfill → dual-read → switch → cleanup.
1 → 1 placement
dual-read window
Решение 9
No-orphan rule
Wish без placement'ов не существует. Триггер при удалении последнего placement: если это был последний — требовать явное full-delete (не auto-cascade), кроме случая каскада от удаления wishlist'а.
enforce invariant
cascade on wishlist delete
Решение 10
Copy — отдельный endpoint
POST /tg/wishes/:id/duplicate-to-wishlists создаёт новые canonical wish записи, каждая — со своим placement. Комментарии и бронь не копируются. Имя: duplicate, не copy — чтобы не пересекаться с shared terminology.
new wish rows
no comment/reservation copy
Решение 11
Not Pro-gated
Shared-placement сам по себе — Free-фича. Paywall касается только количества wishlist'ов: если у Free-юзера 2 лимит, shared работает во все 2.
Free + Pro
paywall ≠ shared
Решение 12
Badge: 🔗 В N
На wish-карточке в списке — мини-pill 🔗 В 3 рядом с title. В detail view — расширенный shared-block с перечислением wishlist'ов. Публичный зритель badge не видит.
inline pill (list)
full block (detail)
