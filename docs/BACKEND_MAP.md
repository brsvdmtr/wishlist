## Business Logic Details

### 1. User Identification

**Telegram Users:**
- Identified by `telegramId` (unique)
- Created/updated on every authenticated request via `getOrCreateTgUser()`
- `telegramChatId` stored for notifications (set by bot /start handler)

**Actor Hash:**
- SHA-256 of Telegram ID, formatted as UUID (8-4-4-4-12)
- Used for anonymous reservation identity
- Generated by `tgActorHash(telegramId)` on server
- Generated by `computeActorHash(telegramId)` on client
- Enables "reserved by me" detection without revealing Telegram ID to others

### 2. Wishlist Lifecycle

```
Create -> Active -> Delete (hard delete, CASCADE)
```
- Owner creates with title + optional deadline
- Slug auto-generated, never changes
- shareToken generated on first share request
- Plan limit: 2 wishlists per user
- Rename via PATCH (title only, slug stays)

### 3. Item Lifecycle `VERIFIED_FROM_CODE`

**Statuses**: AVAILABLE | RESERVED | PURCHASED | COMPLETED | DELETED

```
Create (AVAILABLE) ──→ TG reserve (RESERVED, epoch++, reserverUserId set)
                   ──→ Public reserve (RESERVED, epoch unchanged, no reserverUserId)
RESERVED ──→ TG unreserve (AVAILABLE, reserverUserId=null, comments get 30d TTL)
         ──→ Public unreserve (AVAILABLE, anonymous, no side effects)
         ──→ Public purchase (PURCHASED)
Any ──→ Complete (COMPLETED, owner, comments get 30d TTL)
    ──→ Delete (DELETED, soft delete, owner)
COMPLETED/DELETED ──→ Restore (AVAILABLE, owner)
Admin PATCH: any status → any status (no side effects)
```

**Key difference TG vs Public reserve:**
- TG reserve: sets reserverUserId, increments epoch, creates system comment, sends notification
- Public reserve: only changes status and creates ReservationEvent, no epoch/userId/comments

**Plan limit**: 10 active items per wishlist

### 4. Reservation System

**Reserve flow:**
1. Verify item is AVAILABLE
2. Prisma transaction:
   - Update item: status=RESERVED, reservationEpoch++, reserverUserId
   - Create ReservationEvent (RESERVED, actorHash, displayName)
   - Create SYSTEM comment "Подарок забронирован"
   - Clear TTL on previous epoch comments
3. Notify owner (best-effort Telegram message)

**Unreserve flow:**
1. Verify item is RESERVED
2. Verify actorHash matches (timing-safe comparison)
3. Prisma transaction:
   - Update item: status=AVAILABLE, reserverUserId=null
   - Create ReservationEvent (UNRESERVED)
   - Create SYSTEM comment "Бронь отменена"
   - Set 30-day TTL on all comments
4. No notification sent

**Key design**: Owner NEVER sees who reserved. Only sees item is reserved.

### 5. Comment System `VERIFIED_FROM_CODE`

**Access control:**
- Comments exist **only** via `/tg/items/:id/comments` (Telegram auth)
- **No public comment endpoints** — `/public/items/:id/comments` does NOT exist
- Owner can comment on any reserved item they own
- Reserver can comment on items they reserved
- Third parties get 403 Forbidden
- Archived items (COMPLETED/DELETED) reject new comments

**Anti-spam (5 checks):**
1. 10-second cooldown between comments
2. No duplicate consecutive messages
3. Max 3 unanswered messages in a row
4. Max 10 per hour per author
5. Max 20 per 30 days per author
6. Emoji/dots-only messages rejected

**ReservationEpoch isolation:**
- Each reserve cycle increments epoch
- Comments linked to epoch
- New reserver only sees current epoch comments
- Previous epoch comments anonymized (authorDisplayName masked)

**TTL (time-to-live):**
- 30-day TTL set when item is COMPLETED or UNRESERVED
- Hourly background job deletes expired comments
- TTL cleared when item is re-reserved

**Notifications:**
- First comment: sent immediately
- Subsequent within 30s: batched into "You have N new comments" message
- Keyed by `itemId:recipientUserId`

### 6. Photo System

**Upload:**
- Multer memory storage (buffer)
- Sharp processing: auto-rotate, EXIF strip, resize, JPEG compress
- Two variants: full (1600px, q80) + thumb (480px, q70)
- Filenames: `{uuid}-full.jpg`, `{uuid}-thumb.jpg`
- Stored in UPLOAD_DIR (Docker volume at /data/uploads)
- Old files deleted on replace

**Serving:**
- `express.static('/uploads', UPLOAD_DIR, { maxAge: '30d', immutable: true })`
- Public URL: `https://wishlistik.ru/api/uploads/{uuid}-full.jpg`

**Cleanup:**
- `deleteUploadFile()` deletes both full and thumb variants
- Called on: photo replace, photo delete, item delete (NOT currently - GAP)

### 7. Role Detection (`getItemRole`)

Returns one of three roles:
- **owner**: User.id matches wishlist.ownerId
- **reserver**: Item is RESERVED AND last ReservationEvent actorHash matches (timing-safe)
- **third_party**: Everyone else

Used by: comments endpoints, photo endpoints

### 8. Notification Display Name Bug Fix

When reserver comments, the notification uses:
- `reservationEvents[0].comment` (chosen display name, e.g., "Жопа")
- NOT `req.tgUser.first_name` (real Telegram name, e.g., "Екатерина")

### 9. Sorting

Items sorted by:
1. Active before archived (AVAILABLE/RESERVED/PURCHASED before COMPLETED/DELETED)
2. Priority DESC (HIGH > MEDIUM > LOW)
3. updatedAt DESC
4. createdAt DESC
5. id DESC (stable tiebreaker)

Implemented in both Prisma `orderBy` (ITEM_ORDER_BY) and JS (`sortItemsJs`).
