Wishlist emoji picker — proposed mockups

Текущая реализация: 64px-инпут со смайликом перед названием. Проблемы: непонятно что туда тапать, iOS клавиатура показывает strange auto-fill, смайлик визуально сливается с polygonal-input, нет четкого affordance "это можно изменить".

Ниже два варианта по новой v2.1 ДС. Оба используют примитивы из @wishlist/ui — BottomSheet, TextField, Card. Без iOS-keyboard-emoji пинов и без скрытых affordances.

Вариант A — отдельный picker-sheet recommended

Тап на «обложку» (большая 48×48 плитка с текущим смайлом) открывает BottomSheet с курированной палитрой 21 смайлика + кнопка «Свой» открывает нативный emoji-input. Название редактируется отдельно в стандартном TextField. Чистое разделение задач.

Свой ✎

+ Чистый affordance — большая обложка явно «трогается».
+ Нативная iOS клавиатура не вмешивается.
+ Курированная палитра — выбор за 1 тап в 95% случаев.
+ «Свой» сохраняет гибкость для нестандартных смайлов.

− Дополнительный экран — лишний тап для смены.
− Curated палитра требует поддержки набора (~21 смайлик).

Вариант B — inline expand-tray

60×60 «обложка» с маленьким значком ✎ снизу-справа (явный edit-affordance). Тап сразу разворачивает горизонтальную ленту с быстрыми смайлами + «Еще» — без отдельного экрана. Название и обложка — в одной зоне.

Еще

+ Один экран — никуда уходить не надо.
+ Маленький ✎ значок — нет двойного тапа.
+ Быстрая смена смайлика горизонтальной лентой.

− Лента горизонтального скролла на маленьком экране — менее удобна.
− Чуть теснее визуально (3 элемента в одной строке).
− «Еще» нужен для нативного emoji-picker — лишняя ступень.

Моя рекомендация — Вариант A Sheet-picker даёт лучший visual hierarchy, использует canonical BottomSheet primitive, убирает iOS-emoji-keyboard pain, и курированная палитра решает 95% кейсов одним тапом. Вариант B компактнее, но горизонтальный scroll на 320px ширине — не лучший UX.
