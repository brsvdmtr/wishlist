Правила показа — ВСЕ условия должны быть true

Это guest-view screen === 'guest-view' и guestWl загружен. Viewer открыл чужой вишлист по share-link / slug (self-check #1).
Не owner wishlistsLoaded === true && wishlistCount === 0. Тот же сегмент, что E11 (owner_as_guest). Если у юзера есть свой вишлист — он не ЦА (self-check #2). Гейт на wishlistsLoaded защищает от транзиентного 5xx на /tg/wishlists (иначе owner мелькнёт как гость).
Не показан в этой сессии Memory-flag e13ShownThisSession (ref). Один показ на mount — «dismiss-once per session» из спеки.
Не превышен freq-cap ≤ N показов за 7 дней (rolling). localStorage массив timestamp'ов, прунится до 7 дней. N = 3 по умолчанию (self-check #5).
Нет недавнего dismiss Если dismissed_at < 7 дней назад — skip. Явное закрытие глушит сильнее, чем просто показ.
Priority: 1 banner за раз Если активен birthdayContext-banner (контекстный, сильнее) — E13 в этой сессии не показываем. Митигация UI-clutter из § E13.10. dont_gift-блок информационный (не promo-CTA) и живёт выше — сосуществует.
Эксперимент в treatment useExperiment('e13-guest-banner') === 'treatment'. Control видит чистый список → меряем guardrail (bounce / reservation rate). 50/50. Выключатель — EXP_E13_GUEST_BANNER_ENABLED в .env (можно катить 100% без A/B).
Invariant. E13 — pure inline-контент в потоке списка. Никогда не position: fixed/absolute, не overlay, не modal. Поэтому он структурно не может перекрыть reserve-CTA или любой другой контент (self-check #3).
Guardrail. Primary — G→O D7. Guardrail — guest-view bounce rate и reservation rate: если banner раздражает и гость уходит до резерва, control-арм это поймает. Порог успеха: G→O +≥5%, bounce +≤2%.

Frequency-cap модель

Pure-logic helper — lib/guestBannerCta.ts (зеркало postReservationCta.ts)

// «N раз за 7 дней» = массив timestamp'ов, прунится по окну.
// Явный dismiss = отдельный, более сильный mute.
export const E13_SHOWN_LS_KEY = 'wb_e13_banner_shown_ts_v1';     // number[] (epoch ms)
export const E13_DISMISS_LS_KEY = 'wb_e13_banner_dismissed_at_v1'; // number
export const E13_WINDOW_DAYS = 7;
export const E13_MAX_PER_WINDOW = 3;   // ← N (tunable)
export const E13_EXPERIMENT_KEY = 'e13-guest-banner';
export const E13_ONBOARDING_ENTRY_POINT = 'guest_view_banner';

export function shouldShowGuestBanner(i: E13GateInput): E13Decision {
  if (i.sessionFlag)            return { show: false, reason: 'session_shown' };
  if (i.godModeForce)           return { show: true };
  if (!i.wishlistsLoaded)        return { show: false, reason: 'wishlists_not_loaded' };
  if (i.wishlistCount > 0)       return { show: false, reason: 'owner_as_guest' };
  if (i.birthdayBannerActive)   return { show: false, reason: 'banner_priority' };
  if (i.variant !== 'treatment') return { show: false, reason: 'not_in_treatment' };
  if (i.dismissedAt !== null && i.now - i.dismissedAt < WINDOW_MS)
                                return { show: false, reason: 'dismissed_cooldown' };
  const recent = i.shownTs.filter(t => i.now - t < WINDOW_MS);
  if (recent.length >= E13_MAX_PER_WINDOW)
                                return { show: false, reason: 'freq_cap' };
  return { show: true };
}
Тестируемость. Чистая функция, now инжектится — все 8 gating-условий покрываются unit-тестами без React/DOM/storage. read*/record*-хелперы изолируют localStorage (graceful на quota / private-mode → ведём себя как «ни разу не показан»).
Два уровня тишины. Пассивный показ → impression-cap (N=3 / 7д). Явный × → hard-mute (7 дней целиком). Разные сигналы — разная строгость.

Аналитика

Новые события (supersede backlog'овые g2o.banner_*)

guest_banner.shown Banner попал во viewport (IntersectionObserver, один раз). Props: wishlistId, experimentKey: 'e13-guest-banner', variant: 'treatment', shownCountInWindow (1…N), godModeForce?.
guest_banner.clicked Тап «Создать свой вишлист». Props те же + destination: 'onboarding-entry'. Сразу после — startOnboarding('guest_view_banner') (self-check #4).
guest_banner.dismissed Тап ×. Props: method: 'tap_x', wishlistId. Ставит dismissed_at в localStorage.

Транспорт + allowlist

  • trackEvent(event, props) → буфер → POST /tg/telemetry → AnalyticsEvent (как везде; MiniApp.tsx:4759).
  • Серверный allowlist в routes/telemetry.routes.ts фильтрует по prefix/exact — нужно добавить prefix guest_banner. (как уже есть guest_owner_cta. для E11), иначе события дропнутся.
Funnel. guest_banner.shown → guest_banner.clicked → onboarding_started (entry_point: 'guest_view_banner' — это join-ключ, НЕ wishlist.created.source, который всегда 'miniapp') → guest.converted_to_user (уже wired server-side). Conversion = clicked / shown и created / shown. Один dashboard рядом с E11 — сравниваем passive vs post-reserve канал.

Поведение «Создать свой вишлист»

  • Fires guest_banner.clicked.
  • startOnboarding('guest_view_banner') — новый entryPoint в существующем onboarding-flow (рядом с E11-овским 'post_reservation_claim'). Никакого нового экрана не рисуем — переиспользуем onboarding.
  • Старт онбординга шлёт onboarding_started с entry_point: 'guest_view_banner' (новое значение union EntryPoint).
  • Текущий guest-view не разрушаем деструктивно: юзер уходит в свой onboarding; вернётся к Машиному списку по share-link, если надо.
Не deeplink-возврат. Как и в E11 — после создания вишлиста оставляем юзера в его новом «my-wishlists», а не тащим обратно в чужой список. Activation-момент «теперь ты owner» важнее.

Самопроверка (per task spec)

1. Guest видит bannerFrame 1 — banner в конце guest-view. Гейт screen==='guest-view' && wishlistCount===0.
2. Owner не видит bannerFrame 4 — wishlistCount > 0 → owner_as_guest skip, событие не летит.
3. Banner не перекрывает reserve CTAFrame 2 — reserve-CTA на guest-item-detail, banner на guest-view; разные экраны + inline (не overlay). Invariant выше.
4. Click → own wishlist creation§ Поведение — startOnboarding('guest_view_banner') после clicked.
5. Frequency cap работает§ Freq-cap — N=3 / 7д (impression-cap) + session-flag + 7д dismiss-mute. Frame 3 (dismiss) + Frame 4 (cap reached).

Решения (утверждено Dmitry, 2026-05-30)

  • Размещение — низ списка. Мягче (гость сперва видит ценность всех желаний), нулевой clutter с birthday-banner, ниже bounce-риск. (Backlog § E13.4 предлагал «сверху» — отклонено в пользу мягкости.)
  • N = 3 показа / 7 дней + once-per-session + 7д hard-mute на явный ×.
  • Сразу 100% (не A/B 50/50) через EXP_E13_GUEST_BANNER_ENABLED=true / ROLLOUT=100. Платформенный 5% global holdout всё равно даёт control → фактический reach ~95%. Kill-switch: ENABLED=false + --force-recreate api, без редеплоя.
  • Tone — info (мягкий glass), не promo.
  • Сосуществование с E11 — ОК: разные моменты (passive list vs post-reserve sheet), разные события, без двойного modal.
