E15 · Display-name prefill в reservation flow

Гость открывает чужой вишлист и хочет забронировать подарок. Сейчас имя предзаполняется только из tgUser.first_name. E15 обогащает источник prefill (UserProfile.displayName → TG full name → TG first name), добавляет визуальный индикатор «откуда», и аналитику reservation.display_name_prefilled / reservation.display_name_edited.

Статус: DRAFT 2026-05-25 · Конфликт: нельзя катить параллельно с E14 (оба меняют тот же sheet) · Surprise-mode инвариант: уже залочен apps/api/test/integration/surprise-mode-privacy.test.ts — owner-side эндпоинты не возвращают reservedByDisplayName ни в каком виде.

Аналитические события

EventКогдаProps
reservation.display_name_prefilled При открытии sheet, если prefill источник не none { source: 'profile' | 'tg_full' | 'tg_first', initialLen: number, itemId: string }
reservation.display_name_edited При первом изменении значения относительно prefilled (debounced 500ms или blur) { originalSource, finalLen, clearedFully: boolean, itemId }
Surprise-mode инвариант: оба события — клиентская аналитика гостя, отправляются с его actorHash, не лежат в owner-side эндпоинтах. apps/api/test/integration/surprise-mode-privacy.test.ts продолжает гарантировать, что reservedByDisplayName не утекает в owner DTO.

Что меняется в коде (ориентировочно)

  • 4 точки вызова setGuestName(tgUser?.first_name ?? '') в MiniApp.tsx (17231, 18284, 18299, 18389, 18405) → заменяются на новый хелпер resolveReservePrefill(tgUser, profileData), возвращающий { value, source }.
  • Sheet (~25590-25633): добавляется prefill-hint + reset-link + edited-state детекция.
  • Новый state: guestNamePrefillSource (для сравнения «изменено или нет») и guestNameEditedFired (чтобы не дублировать событие display_name_edited на каждое нажатие).
  • 5 локалей в packages/shared/src/i18n.ts: 4 новых ключа.
  • API не трогаем. POST /tg/items/:id/reserve уже принимает displayName, контракт не меняется. Прироста payload нет.
Конфликт с E14: E14 (anonymous reserve) меняет тот же sheet — нельзя катить параллельно. В surprise-mode-privacy.test.ts для E14 уже стоит it.todo. Порядок: сначала E15, потом E14 поверх.
Out of scope — secret reservation: второй CTA на item screen (sr_cta_reserve_secretly) открывает отдельный confirm sheet и шлёт POST /tg/items/:id/secret-reserve с payload { note } — без displayName. Secret-бронь невидима ни владельцу, ни другим гостям (item остаётся AVAILABLE для всех, кроме резервирующего) — аудитории для display name просто нет. E15 затрагивает только public reserve flow.

План тестов (по требованию задачи)

  • Unit — resolveReservePrefill(): 4 ветки (profile / tg_full / tg_first / none) + крайние случаи (пустые строки, whitespace, длина >64).
  • Component (RTL) — открытие sheet с разными комбинациями tgUser + profileData: проверяем (a) значение в input, (b) text hint, (c) trackEvent вызывается с правильным source.
  • Component (RTL) — изменение значения: событие display_name_edited шлётся один раз, повторные нажатия не плодят дубликаты; кнопка «Вернуть» восстанавливает prefilled value.
  • Integration (existing) — surprise-mode-privacy.test.ts остаётся зелёным; добавляем кейс: после POST /reserve с edited значением, GET owner-side эндпоинты по-прежнему не возвращают reservedByDisplayName.
  • API smoke — reservation.succeeded событие продолжает шлёться (текущий код по-прежнему вызывает trackAnalyticsEvent на line 1310).
